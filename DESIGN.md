# Infinite Canvas Studio 设计规范

本文件把用户提供的桌面界面参考、冻结上游的项目工作台行为和目标 React/shadcn 实现统一为后续 UI 开发的强制设计基线。

## 1. Design Specification

### Purpose Statement

Infinite Canvas Studio 是创作者长时间停留的本地桌面工作区。界面必须让项目、画布、素材、模型与任务始终保持空间关系，减少页面跳转和装饰性信息，让用户把注意力留在内容与画布上。

### Aesthetic Direction

**Industrial / utilitarian，brutally minimal。** 视觉来自专业创作工具和制图工作台：清晰分区、细线框、硬朗黑色操作面、克制的状态色和可无限延伸的网格。不是营销页、柔和消费应用，也不是带彩色渐变的通用 SaaS Dashboard。

### Color Palette

主结构只使用中性颜色；品牌与状态色保持窄用途。

| 语义         | 浅色主题  | 深色主题  | 用途                       |
| ------------ | --------- | --------- | -------------------------- |
| `background` | `#FFFFFF` | `#10141D` | 应用栏、上下文面板         |
| `workspace`  | `#F6F7F9` | `#111722` | 无限工作面                 |
| `surface`    | `#FFFFFF` | `#151B26` | 卡片、弹层、输入           |
| `foreground` | `#151922` | `#F4F7FB` | 主文字、主图标             |
| `muted`      | `#667085` | `#AAB5C4` | 次级文字、非活动图标       |
| `border`     | `#E5E9EF` | `#2D394C` | 1px 分隔与控件边界         |
| `primary`    | `#050505` | `#F4F7FB` | 主按钮、当前导航、当前项目 |
| `smart`      | `#2563EB` | `#60A5FA` | 智能画布、选择、可交互链接 |
| `danger`     | `#FF3347` | `#FF5A6B` | 删除、错误、回收站计数     |

禁止紫色、靛青或蓝紫渐变作为默认“AI”表达。供应商品牌色只能出现在其标识和受控状态标签内。

### Typography

- 拉丁、数字与技术标识：`Geist Variable`。
- 中文：`Noto Sans SC`；发布包必须本地携带所需字重或使用经批准的等宽替代，不依赖在线字体。
- 正文字重只用 400/500/600，标题与关键操作最多 700；禁止依赖非标准 `850` 等浏览器插值权重。
- 数据、快捷键、哈希和日志可用 `Geist Mono`。
- 默认正文 13px，辅助信息 11–12px，面板标题 15–16px，页面标题 20–24px。桌面工具界面不使用营销式 40px+ 标题。

### Layout Strategy

采用非对称、可折叠的三层桌面结构：

1. **应用栏 / 导航轨**：固定在最左侧，紧凑态只显示图标，展开态显示中文标签。
2. **上下文面板**：根据当前模块展示项目、素材分类、会话、工具或设置导航。
3. **无限工作面**：占据剩余区域，顶部仅保留当前上下文和高频操作，内容可平移/缩放。

项目卡片位于世界坐标而非规则卡片网格中；它们可以形成不对称空间排布。大面积留白是画布的工作空间，不是居中内容的装饰背景。

## 3. 桌面壳层尺寸

以下为 100% UI 缩放下的目标区间，可由可访问性缩放整体调整：

| 区域         | 目标尺寸              | 规则                                         |
| ------------ | --------------------- | -------------------------------------------- |
| 自定义标题栏 | 40–44px 高            | 左侧平台窗口区，标题不可与操作按钮争抢空间。 |
| 紧凑导航轨   | 72–80px 宽            | 44px 点击目标，图标居中。                    |
| 展开导航     | 232–256px 宽          | 图标、标签和活动背景同一行。                 |
| 上下文面板   | 272–320px 宽，可调整  | 默认项目面板约 288px。                       |
| 工作面顶栏   | 60–64px 高            | 标题在左，高频动作在右。                     |
| 项目/画布卡  | 248–288px × 140–156px | 世界坐标定位，最小 24px 间距。               |
| 图标按钮     | 36–40px               | 可见 1px 边框，必须有 tooltip/aria-label。   |
| 主操作按钮   | 36–40px 高            | 黑底白字；图标在前，文字简短。               |

最小支持窗口为 1024×720。窗口变窄时先折叠导航标签，再缩窄上下文面板；不把桌面壳层重排成移动端单列。上下文面板可以由用户隐藏，但当前模块与恢复入口必须始终可达。

## 4. 结构与组件

### 4.1 标题栏

- 展示原项目图标和“无限画布”或当前文档名称。
- 只有明确标记的空白区域可拖拽；按钮、输入、菜单必须 `no-drag`。
- 不在标题栏放业务级工具栏；同步、任务、更新等状态使用右侧克制图标。

### 4.2 导航轨

固定顺序：画布列表、在线生图、对话、工具箱、素材、设置。底部放语言状态和主题切换；`1.x` 只提供中文，语言项作为只读状态或隐藏，不制造不可用入口。

- 紧凑态显示 Lucide 图标与 tooltip。
- 展开态显示图标、中文标签和整行活动背景。
- 活动态使用接近黑色的矩形块，圆角不超过 4px；不使用彩色发光。
- 展开/收起 160–220ms，宽度和标签透明度同步；尊重 `prefers-reduced-motion`。

### 4.3 上下文面板

- 面板标题、添加按钮、滚动列表和固定底栏形成明确四段。
- 当前项目使用黑底白字；数量使用紧凑方形/小圆角计数器。
- 重命名、删除等低频动作在 hover、焦点或上下文菜单中出现，不长期占据列表宽度。
- 回收站固定在底部；危险计数使用红色，不用红色装饰普通状态。

### 4.4 工作面与卡片

- 背景叠加 24px 点阵和 120px 主网格；浅色线透明度低于 10%，不能干扰节点连线。
- 鼠标/触控板支持平移、缩放、适配视图；工作面空白处双击可进入新建流程时必须能撤销或取消。
- 卡片使用白色/深色实面、1px 边框和 0–4px 基础圆角。常态几乎无阴影，hover/拖拽才提升阴影。
- 卡片信息顺序：类型标签 → 标题 → 节点数/更新时间；菜单固定右上角。
- “智能画布”使用 `smart` 蓝色标签，“传统画布”使用中性标签。标签颜色不改变卡片主体。
- 删除必须先进入显式确认状态；被剪切项使用降低透明度和虚线边框。

### 4.5 表单、弹层与反馈

- 优先使用行内编辑、侧栏或靠近触发点的 popover；复杂设置才使用 dialog。
- 圆角层级：基础 0–4px，控件 4–6px，弹层最多 8px，计数/状态胶囊可用全圆角。
- 阴影层级只允许 `overlay`、`dragging`、`focus-elevated` 三类，普通布局不使用大面积浮层阴影。
- 空、加载、失败、离线、权限不足、资料库缺失和 sidecar 故障都要有专门状态；禁止只显示空白或无限 spinner。

## 5. 交互与状态

- 桌面一级操作要有稳定快捷键，并通过命令面板可发现。
- hover 不是唯一入口；所有 hover 动作必须能通过键盘焦点或上下文菜单访问。
- 画布选择、框选、拖拽与连线使用 60fps 优先的 transform，避免持续修改布局属性。
- 自动保存不弹成功 toast；仅在延迟、失败、离线或恢复时显示状态。
- 任务提交后在节点和任务中心同时可追踪。失败不自动重试，错误信息必须给出下一步。
- 动画使用 120–220ms；只动画 opacity、transform 和必要的面板宽度。禁止装饰性循环动画。

## 6. shadcn/ui 使用边界

shadcn/ui 是可访问交互原语，不是默认视觉模板：

- 可以使用 Button、Dialog、Popover、Tooltip、DropdownMenu、Tabs、ScrollArea、Command 等原语。
- 不把每个内容区包装成默认 `Card`，不使用居中 Hero 和“三张卖点卡”的脚手架布局。
- 组件变体映射到本文件令牌；页面组件不能自行引入紫色渐变、夸张圆角或无语义阴影。
- 领域组件命名表达用途，例如 `ProjectRail`、`CanvasBoardCard`、`TaskStatusStrip`，而不是 `PrettyCard` 或 `ModernSidebar`。

## 7. 无障碍与平台约束

- 常规文字与背景至少满足 WCAG 2.2 AA；焦点环在浅/深主题均清晰。
- 图标按钮具有中文可访问名称和 tooltip；状态不能只靠颜色表达。
- 支持键盘遍历导航、项目列表、顶栏和画布辅助列表；拖拽操作提供菜单或快捷键替代。
- macOS 使用 `⌘`，Windows 使用 `Ctrl`；文案和快捷键提示按平台渲染。
- 200% UI 缩放下仍能访问主要操作，允许上下文面板滚动，不裁掉确认按钮。

## 8. 视觉自审

每次 UI 交付至少检查：

- 是否仍是三层桌面工作区，而不是通用网页布局。
- 是否只使用批准的语义色、字体、圆角和阴影层级。
- 是否同时覆盖导航紧凑/展开、浅色/深色、空/加载/错误状态。
- 是否无 emoji 图标、无紫色 AI 渐变、无大面积居中卡片陈列。
- 是否在目标窗口尺寸和 200% 缩放下可操作。
- 是否用实际 Electron 窗口截图与 `docs/reference/ui/` 并排评审。

当前 `apps/renderer/src/App.tsx` 是工程连通性占位页，不是获准的产品首页；后续第一个 UI 纵向切片应以本规范重建桌面壳层。
