# 实现对齐说明

本文回答后续开发最容易混淆的问题：哪些内容要与上游一致，哪些内容以用户参考图为准，哪些内容必须按新架构重写。

## 1. 三轴对齐合同

| 轴   | 权威来源                                 | 必须保持                                                 | 允许改变                                    |
| ---- | ---------------------------------------- | -------------------------------------------------------- | ------------------------------------------- |
| 功能 | 冻结上游 + 功能矩阵                      | 用户可见能力、输入输出、错误与恢复结果                   | 页面组织、API URL、内部模型、实现语言       |
| 设计 | 用户参考图 + `DESIGN.md`                 | 信息架构、密度、层级、黑白工具感、紧凑/展开导航          | 精确像素、上游旧字体、临时动画细节          |
| 架构 | `TECHNICAL_ARCHITECTURE.md` + 已批准决策 | 本机 SQLite、Python 单一所有者、进程安全边界、版本化契约 | 上游 JSON、巨型 FastAPI 文件和直接 DOM 脚本 |

当三轴冲突时，不做“折中猜测”：先按 `AGENTS.md` 的来源优先级裁决并记录差异。

## 2. 桌面信息架构映射

| 目标入口 | 上游证据                                                        | 参考图含义                             | 目标实现                          | 主要矩阵                                       |
| -------- | --------------------------------------------------------------- | -------------------------------------- | --------------------------------- | ---------------------------------------------- |
| 画布列表 | `static/canvas-list.html`、`static/js/canvas-list.js`           | 导航活动态、项目面板、世界卡片、回收站 | renderer 壳层 + projects 模块     | `IC-SHELL-010`、`IC-PROJECT-*`                 |
| 在线生图 | `angle.html`、标准/智能画布的直接生成入口                       | 直接选择供应商/模型并生成的唯一主入口  | generation 工作区 + provider 服务 | `IC-NODE-005/006`、`IC-PROVIDER-*`             |
| 对话     | `gpt-chat.html`、`/api/chat`、`/api/conversations`              | 独立上下文面板与主内容面               | conversations 模块 + 流式事件     | `IC-CHAT-*`、`IC-NODE-004`、`IC-TASK-*`        |
| 工具箱   | `enhance.html`、`klein.html`、`zimage.html`、LTX 与历史批量工具 | 增强和专用工作流的唯一主入口           | 领域工具注册表，不做 iframe 集合  | `IC-EDIT-*`、`IC-COMFY-*`、`IC-NODE-*`         |
| 素材     | `asset-manager.html/js/css`                                     | 左侧分类 + 高密度媒体工作区            | assets 模块 + 内容寻址存储        | `IC-ASSET-*`                                   |
| 设置     | `api-settings.html`、`comfyui-settings.html`                    | 左侧设置分类 + 右侧配置面板            | providers/comfy/settings 模块     | `IC-PROVIDER-*`、`IC-COMFY-*`、`IC-SETTINGS-*` |

导航名称来自参考图，是产品信息架构；上游的 13 个 HTML 页面不会一对一变成 13 个 Electron 窗口。
一级入口拥有唯一主路由：`在线生图` 不承载 Enhance、Klein、Z-Image 或 LTX；`工具箱` 不复制直接供应商生成页。画布节点、搜索结果和任务历史可以深链到这些主路由，但不得建立第二套状态或页面实现。

## 3. 项目工作台详细对齐

上游锚点：

- DOM：`static/canvas-list.html`。
- 视觉：`static/css/canvas-list.css`。
- 行为：`static/js/canvas-list.js`。
- API：`main.py` 中 `/api/projects`、`/api/canvases` 与回收站路由。

目标行为：

| 能力     | 上游行为                                       | 目标差异/约束                                       | 验收                       |
| -------- | ---------------------------------------------- | --------------------------------------------------- | -------------------------- |
| 项目列表 | 创建、重命名、删除、计数、选择                 | 结构化数据进入本机 SQLite；默认项目不可误删         | 项目 CRUD + 重启恢复       |
| 画布看板 | 卡片为世界坐标，支持平移、缩放、拖拽和重置视图 | React 组件/画布状态实现，不直接移植 DOM 脚本        | 位置/视口持久化 + 交互 E2E |
| 新建画布 | 支持传统/智能画布并可在世界坐标创建            | 类型必须映射版本化 canvas schema                    | 创建后可打开、保存、恢复   |
| 画布管理 | 打开、重命名、移动项目、剪切粘贴、删除         | 命令进入操作日志；跨模块调用通过应用服务            | 正常/取消/失败/撤销测试    |
| 画布导出 | 菜单分别提供“导出画布”与“导出画布 + 资源”      | 统一映射到版本化 `.icanvas`，资源为可选打包项       | 格式契约 + E2E             |
| 回收站   | 恢复和彻底删除；上游文案为 30 天               | 已批准产品规则覆盖为 7 天；NAS 实体不可自动物理删除 | 时间控制 + 引用安全测试    |
| 卡片信息 | 类型、标题、节点数、更新时间、菜单             | 视觉采用参考图的更硬朗、低圆角样式                  | 视觉对照 + 键盘访问        |

## 4. 上游行为到目标模块

| 上游区域                    | 不直接复用的原因                 | 目标所有者                                        |
| --------------------------- | -------------------------------- | ------------------------------------------------- |
| `main.py` 152 个路由/Socket | 业务、持久化、协议耦合在巨型文件 | FastAPI `api/` + 各领域应用服务                   |
| `static/js/canvas.js`       | DOM、状态、网络和画布算法耦合    | `packages/canvas-model` + renderer canvas feature |
| `static/js/smart-canvas.js` | 与标准画布重复且依赖全局状态     | 共享画布模型 + 可组合智能节点/运行器              |
| `static/js/canvas-list.js`  | 直接 DOM/fetch/localStorage      | projects API client + React state + SQLite        |
| `asset-manager.js`          | 文件索引与 UI 混合               | backend assets 模块 + renderer asset feature      |
| `api-settings.js`           | 供应商协议与表单耦合             | provider registry/adapters + settings UI          |
| `comfyui-settings.js`       | 实例探测、工作流和 UI 混合       | comfy adapter/scheduler + workflow UI             |
| Chrome/Photoshop 现有组件   | 旧接口与安全边界需重做           | 版本化配对协议 + 独立能力令牌                     |

## 5. 已知差异与目标新增

| 上游                                   | 目标                                   | 追踪方式                                                  |
| -------------------------------------- | -------------------------------------- | --------------------------------------------------------- |
| JSON/文件式结构化数据                  | 本机 SQLite + Alembic                  | 已批准重构                                                |
| 可能从局域网访问 Web 服务              | 只监听 `127.0.0.1`                     | 已批准安全边界                                            |
| 在线人数                               | 删除，只保留任务/素材事件              | `IC-LEGACY-001` 批准差异                                  |
| 中英文切换                             | `1.x` 只提供中文                       | `IC-SHELL-011` 目标要求                                   |
| 应用内 Web 更新流程                    | 检查 Release 后由系统浏览器打开        | `IC-UPDATE-001` 目标要求                                  |
| 回收站页面写 30 天                     | 产品规则为 7 天                        | `IC-ASSET-005` 目标要求                                   |
| 上游 Inter/Space Grotesk 与较软圆角    | Geist + Noto Sans SC，工业工具型低圆角 | 用户参考图与 `DESIGN.md`                                  |
| 传统页面式导航                         | 三层桌面壳层与上下文面板               | `IC-SHELL-010` 目标要求                                   |
| 项目仅含创建、重命名、删除、计数、选择 | 项目复制、恢复与搜索                   | `IC-PROJECT-001` 目标新增；非上游复刻，实现时单独验证来源 |

## 6. 实现任务模板

每个后续功能任务至少记录：

```text
矩阵编号：IC-...
用户目标：...
上游锚点：文件 + 函数/DOM/路由
参考图/设计条款：DESIGN.md 章节
目标模块：renderer / desktop / backend / contracts / extension
契约或 schema 变化：...
正常路径：...
失败与恢复路径：...
自动测试：...
人工/双平台证据：...
明确差异：无 / 已批准条目
```

没有上游锚点或验收证据的“完成”声明无效。

## 7. 当前对齐结论

- 产品范围与会话决策已经由 PRD、技术架构和矩阵承接，没有发现需要回退到上游旧架构的理由。
- 参考图与上游 `canvas-list` 的核心空间模型一致：项目上下文 + 可平移世界卡片；参考图进一步规定了新的桌面导航层。
- 当前仓库骨架符合目标进程边界，但业务实现仍几乎为空；“骨架存在”不等于功能复刻。
- 下一个可信里程碑应是项目工作台纵向切片，而不是继续增加静态占位页。
